# AGENTS

Рабочий контекст для этого проекта.

## Цель файла

Этот файл нужен, чтобы любой следующий агент быстро понял:

- где лежит актуальная информация о геймплее;
- где смотреть список перков;
- где вести backlog идей и задач;
- какие файлы нужно синхронизировать после изменений.

## Главные источники правды

### Код

- [`src/game.js`](./src/game.js)
  Главная игровая логика, баланс, UI на canvas, перки, опасности, движение, генерация мира.

- [`styles.css`](./styles.css)
  Основные стили игры и оверлеев.

- [`index.html`](./index.html)
  Основная страница игры и DOM-оверлеи.

### Документация

- [`gameplay.md`](./gameplay.md)
  Общее описание геймплея, систем и правил ранa.

- [`scrap-perks.md`](./scrap-perks.md)
  Полный список scrap-перков, их эффекты, прогрессия и потолки.

- [`manual.html`](./manual.html)
  Игровой мануал для игрока.

- [`scrap-perks.html`](./scrap-perks.html)
  Отдельная страница для быстрого ревью scrap-перков.

### Рабочий backlog

- [`TODO.md`](./TODO.md)
  Сюда складываются идеи, баги, заметки по балансу и интерфейсу.

## Как работать с TODO

Если по ходу работы появляются:

- идеи;
- баги;
- спорные UX-моменты;
- балансные заметки;
- отложенные улучшения;

их нужно коротко записывать в [`TODO.md`](./TODO.md), а не держать только в памяти диалога.

Правило простое:

- срочное и ближайшее -> `Inbox`
- позже -> `Later`
- сырые идеи -> `Maybe`

Не надо превращать `TODO.md` в большой design doc. Это именно рабочий список.

## Когда надо обновлять docs

После изменений в логике игры надо проверять, не устарели ли:

- [`gameplay.md`](./gameplay.md)
- [`scrap-perks.md`](./scrap-perks.md)
- [`manual.html`](./manual.html)
- [`scrap-perks.html`](./scrap-perks.html)

Минимальное правило:

- если изменился scrap-перк -> обновить [`scrap-perks.md`](./scrap-perks.md)
- если изменился core gameplay / цель / ресурсы / опасности / карта / база -> обновить [`gameplay.md`](./gameplay.md)
- если изменилось то, что видит игрок в help/manual -> обновить [`manual.html`](./manual.html)

## Debug и служебные файлы

Есть debug-артефакты, которые обычно не надо коммитить без явной причины:

- `debug/map-seed-*.svg`
- `debug/runtime-check.png`

Они полезны локально, но не должны автоматически попадать в каждый коммит.

## Проверки после правок

Минимальная проверка после изменений в JS:

```bash
node --check src/game.js
```

Если менялась debug-генерация:

```bash
node --check scripts/render-map-debug.js
```

Если менялись docs или HTML/CSS, проверь:

- не появились ли устаревшие названия перков;
- не осталось ли старых чисел после ребаланса;
- не расходятся ли `manual`, `gameplay` и `scrap-perks`.

## Что важно помнить про проект

- Игра быстро меняется; docs легко устаревают.
- Самый частый источник рассинхрона — `scrap-perks.md` и `manual.html`.
- Пользователь любит быстро итерировать баланс, UI и механику.
- Если что-то явно спорное или недоделанное, лучше записать это в [`TODO.md`](./TODO.md), чем потерять.

## Практический порядок действий для следующего агента

1. Прочитать [`TODO.md`](./TODO.md)
2. Прочитать релевантные разделы [`gameplay.md`](./gameplay.md)
3. Если задача про scrap-перки, открыть [`scrap-perks.md`](./scrap-perks.md)
4. Проверить реальное состояние логики в [`src/game.js`](./src/game.js)
5. После правок синхронизировать docs
6. Если появилась новая идея или отложенная задача, записать ее в [`TODO.md`](./TODO.md)
